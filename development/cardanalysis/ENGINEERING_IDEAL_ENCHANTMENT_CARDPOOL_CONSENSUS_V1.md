# Engineering Ideal Enchantment Cardpool Consensus V1

## Purpose

记录当前关于 `理想` 承载方式与 `工科` 职业卡组方向的阶段性共识。

本文档不是最终卡牌表，不写运行时数据，不决定平衡数值。它的作用是防止后续设计再次滑回
`75 * 5` 卡池乘法、理想限制选牌、隐藏道德审判或额外资源条膨胀。

## Current Design Sentence

```text
职业卡组负责每回合好玩；
理想附魔负责把牌组染上人格和价值倾向；
压力爆表后的美德/折磨负责改变这些倾向的表现。
```

工科职业的当前核心句：

```text
工科：把混乱现实转化成可运行系统。
```

## Ideal Carrier Consensus

`理想` 当前推荐承载方式是 `通用附魔模板`。

| 项 | 当前共识 |
| --- | --- |
| 理想定位 | 不再作为职业、不再作为大卡池、不限制玩家选牌。 |
| 承载方式 | `理想 = 通用附魔模板`。 |
| 理想数量 | 保留 5 色理想。 |
| 模板数量 | 每色约 2-4 个通用附魔，第一版总量优先控制在 10-15 个。 |
| 卡牌关系 | 职业/学科卡牌是主体；理想通过附魔贴到卡牌上。 |
| 附魔上限 | 第一版默认每张牌最多 1 个理想附魔。 |
| 理想强度 | 由牌组中某色附魔数量、质量和关键牌自然形成，不额外做复杂理想等级。 |
| 美德关系 | 美德触发后强化主导理想色附魔，或让该色附魔短期升华。 |
| 折磨关系 | 折磨触发后污染主导理想色附魔，或让该色附魔带副作用。 |

### Explicit Avoid List

| 不做 | 原因 |
| --- | --- |
| `75 * 5` 理想卡牌乘法 | 内容量、平衡量和玩家理解成本失控。 |
| 每张牌手写五色版本 | 会伪装成附魔系统，但实际回到卡池乘法。 |
| 理想限制玩家选牌 | 容易变成隐藏审判或道德扣分，而不是构筑乐趣。 |
| 每回合隐藏统计玩家行为是否符合某理想 | 反馈不透明，玩家会觉得系统在审判自己。 |
| 复杂理想等级树 | 容易和卡牌构筑抢主导权，也会抬高心智负担。 |

## Engineering Cardpool Consensus

工科不按真实下级学科平铺。真实学科名称用于卡名、事件、论文题目、创新点、附魔气质和美术表达；
职业卡组按机制轴组织。

第一版卡池建议规模：

| 项 | 建议 |
| --- | --- |
| 职业牌总量 | 40-50 张。 |
| 玩家可记住的主轴 | 最多 3 个。 |
| 通用攻防/减压/抽滤 | 约 12-16 张，保证任何构筑可玩。 |
| 主轴 A | 约 10-14 张，可独立成型。 |
| 主轴 B | 约 10-14 张，可独立成型。 |
| 主轴 C | 约 8-12 张，最好能与 A/B 交叉。 |
| 稀有拐点/跨轴牌 | 约 4-6 张，用于改变构筑方向。 |

## Engineering Three Axes

当前工科职业三轴为：

| 轴 | 对应学科气质 | 核心体验 | 是否依赖数据 |
| --- | --- | --- | --- |
| 数据方法 | 信息、统计、仪器、测绘、计算机 | 生成、强化、清洗、消耗临时数据牌。 | 是 |
| 自动化方法 | 控制、机械、电气、化工流程、软件工程 | 部署装置/流程，让后续回合自动触发收益。 | 否 |
| 经验方法 | 力学、材料、土木、安全、可靠性、工程现场经验 | 从问题、返工、异常案例中获得可靠性和收益。 | 可交叉，不强依赖 |

三轴的功能句：

```text
数据方法让你看懂问题；
自动化方法让系统自己运转；
经验方法让失败也变成资产。
```

## Axis A: Data Method

### Core Rule

`数据` 不是资源条，而是一组随机生成的 0 费临时牌。

| 项 | 当前共识 |
| --- | --- |
| 费用 | 0 费。 |
| 生命周期 | 临时生成，回合结束消失。 |
| 种类 | 多种数据牌，每张有不同小效果。 |
| 玩法 | 生成数据、强化数据、清洗数据、消耗手牌中的数据获得收益。 |
| 类比 | 有点像猎人的小刀，但每张数据不同，不只是伤害。 |
| 明确不做 | 不做 `记录`、`数据点`、`已记录数据` 等额外计数资源。 |

### Data Card Pool Draft

第一版数据牌种类应克制，建议 5-7 种。

| 数据牌 | 0 费效果方向 | 气质 |
| --- | --- | --- |
| 样本数据 | 抽 1，然后弃 1。 | 普通实验结果。 |
| 对照数据 | 获得少量抗压。 | 有基准，心里稳。 |
| 仿真数据 | 下一张牌数值提高。 | 模型预测。 |
| 异常数据 | 给敌人负面或提高收益，但自身压力小幅增加。 | 有价值但危险。 |
| 噪声数据 | 无效果，主要等待清洗或被消耗。 | 数据污染。 |
| 负结果 | 小幅减压；若被解释或消耗可获得额外收益。 | 失败也有信息量。 |

### Data Subdirections

| 子方向 | 玩法说明 |
| --- | --- |
| 数据质量 | 清洗噪声、强化某张数据、指定或提高数据生成质量。 |
| 数据产量 | 一次生成更多随机数据，承担噪声或压力风险。 |
| 数据消耗 | 消耗手牌中的数据牌，触发模型、论文图表、预测收益。 |

示例名称仅作方向记录：

| 名称 | 作用方向 |
| --- | --- |
| 批量采样 | 生成多张随机数据，可能混入噪声。 |
| 数据清洗 | 将噪声变为样本，或强化一张非噪声数据。 |
| 回归模型 | 消耗手牌中的数据牌，根据数量造成收益。 |
| 论文图表 | 若本回合打出过多种数据，抽牌并减压。 |

## Axis B: Automation Method

### Core Rule

`自动化方法` 必须能不依赖数据独立成型。它负责工科的系统搭建感，而不是数据流的附属模块。

| 子方向 | 玩法说明 |
| --- | --- |
| 装置 | 能力牌或持续状态，部署后每回合触发。 |
| 流水线 | 每打出第 N 张牌、每回合第一次某类牌等条件触发收益。 |
| 反馈控制 | 根据压力、抗压、手牌或回合状态自动修正局面。 |

示例名称仅作方向记录：

| 名称 | 作用方向 |
| --- | --- |
| 自动化脚本 | 每回合第一次打出技能牌时抽牌。 |
| 控制回路 | 回合结束时若有抗压则减压。 |
| 机械联动 | 每打出第 3 张牌造成少量伤害或获得抗压。 |
| 流水线部署 | 每回合开始随机降低一张手牌费用。 |
| 稳态运行 | 若已部署多张能力/装置，获得额外抗压或减压。 |

## Axis C: Experience Method

### Naming

此轴早期被称为 `失效工程`。当前更推荐对玩家展示为 `经验方法`。

| 名称 | 评价 |
| --- | --- |
| 失效工程 | 准确但偏硬，听起来负面。 |
| 经验方法 | 更婉转，适合研究生/大学生玩家理解。 |
| 可靠性方法 | 工科味强，但略专业。 |
| 试错方法 | 直观，但略口语和单薄。 |

### Core Rule

`经验方法` 不是单纯惩罚轴。它表达的是：经历问题、修正问题、从问题中获得可靠性。

| 子方向 | 玩法说明 |
| --- | --- |
| 极限工况 | 用压力、问题牌或下回合不稳定性换取当前高收益。 |
| 安全余量 | 抵消压力、延缓爆表、降低问题牌影响。 |
| 案例复盘 | 消耗或转化问题牌，获得抽牌、减压、成长或伤害。 |

### Negative Card Naming

内部可暂称 `故障`，但玩家展示可更婉转。

| 直接名 | 可展示名 | 适用场景 |
| --- | --- | --- |
| 故障 | 问题样本 | 通用负面集合。 |
| 技术债 | 技术债 | 软件、自动化、赶工相关。 |
| 异常案例 | 异常案例 | 经验方法与数据方法交叉。 |
| 返工 | 方案调整 | 需要消耗/修正的状态牌。 |

示例名称仅作方向记录：

| 名称 | 作用方向 |
| --- | --- |
| 极限工况 | 获得强收益，但增加压力或加入问题牌。 |
| 强行上线 | 高伤害，加入问题牌或技术债。 |
| 安全余量 | 防止一次压力爆表或抵消问题牌副作用。 |
| 根因分析 | 消耗问题牌，抽牌并减压。 |
| 案例复盘 | 每当消耗问题牌，强化一张牌或获得长期收益。 |

## Cross-Axis Hooks

三轴应能交叉，但不要互相强绑。

| 交叉 | 方向 |
| --- | --- |
| 数据方法 + 自动化方法 | 每回合第一次生成数据时，自动化装置触发抽牌、减费或抗压。 |
| 数据方法 + 经验方法 | 异常数据、负结果或噪声可用于定位问题牌。 |
| 自动化方法 + 经验方法 | 装置提供稳定收益；问题牌可能干扰装置，也可被装置自动修正。 |
| 三轴交汇 | 用数据发现问题，用自动化处理流程，用经验方法把问题变成资产。 |

## Relationship To Ideals

工科三轴是职业卡组内容；理想附魔是跨职业的价值倾向层。二者不应互相替代。

| 层 | 职责 |
| --- | --- |
| 工科卡组 | 提供数据、自动化、经验方法三种可玩的职业机制。 |
| 理想附魔 | 给工科牌染上价值倾向，例如更守护、更求真、更荣誉、更自由、更生存。 |
| 压力 | 提供进入戏剧阈值的主血条。 |
| 美德/折磨 | 在压力爆表时强化或污染主导理想色附魔，并制造戏剧节点。 |
| 创新点 | 论文发表后的中高强度构筑奖励，强于普通精英奖励但通常低于 Boss 奖励。 |

示例：

| 工科轴 | 可被理想附魔改变的方式 |
| --- | --- |
| 数据方法 | 求真附魔让数据更干净；荣誉附魔让异常数据更有爆发；守护附魔让对照数据更能减压。 |
| 自动化方法 | 秩序/守护附魔让装置更稳定；自由附魔让流程更灵活；成长附魔让装置随回合增强。 |
| 经验方法 | 生存/守护附魔让问题修复更安全；野心/荣誉附魔让极限工况收益更高但代价更明显。 |

## Open Questions

| 问题 | 当前倾向 |
| --- | --- |
| 三轴名称是否最终锁定为 `数据方法 / 自动化方法 / 经验方法`？ | 当前推荐，但可在写正式卡牌前再命名微调。 |
| 数据牌种类多少合适？ | 第一版 5-7 种，优先少而清楚。 |
| 数据牌是否允许进入弃牌堆？ | 当前倾向为临时牌，回合结束消失；若消耗/弃置互动需要，可只在本回合内处理。 |
| 经验方法负面集合正式名叫什么？ | 内部暂称故障，展示名优先考虑问题样本、异常案例、技术债。 |
| 自动化方法是否需要类似 `集中` 的专属增强值？ | 暂不引入。优先用能力牌和触发条件表达，避免新增资源条。 |

## Minimal Next Slice

下一步可以只做纸面卡池骨架，不急着实现：

```text
1 个工科职业
3 条机制轴
40-50 张职业牌骨架
5-7 种临时数据牌
每条轴 2-3 张代表卡
每色理想 2 个附魔模板
1 个美德强化附魔示例
1 个折磨污染附魔示例
```

最小验证问题：

```text
不依赖完整卡池时，
数据方法是否已经像随机证据而不是第二能量？
自动化方法是否能不吃数据也独立好玩？
经验方法是否让失败变成资产，而不是纯惩罚？
理想附魔是否能改变牌组气质，而不是制造卡池乘法？
```
