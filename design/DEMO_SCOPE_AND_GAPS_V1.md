## 1. Demo 目标（你可以改）

- **体验时长**：大约 30–60 分钟，可在一两局内完整体验一条小循环。
- **核心内容**：
  - 一条精简的 **战役甘特路径**（从开局 → 数场战斗/事件 → 结尾）。
  - 至少 3–5 个有特色的 **敌人**（包含 1 个精英/小 Boss）。敌人设计了5个，只适配一个战斗，其他的战斗场景还没有做。 boss没做
  - 一套可玩的 **卡组 + 白色扩展**（已经基本具备）。
  - 1 次可感知的 **组会/Meeting**（升级/删牌/购卡 + 1–3 条事件）。组会有事件了，但是很粗制滥造
  - 几个 **生活梗事件**（气泡或对话形式，让玩家看到博士日常）。 有了，但是很粗制滥造
- **调性**：以 **“扎心 + 爽 + 反思”** 为主，发疯度控制在 0–1 之间，少量 2 级作为彩蛋。

## 3. 面向 Demo 的关键差距（精简版 TODO，大方向）

> 这里只列“为了一个可玩的 Demo 必须补上的坑”，不是全项目 TODO。  
> 你可以直接在每条后面打勾、补充子任务。

- **Gap 1：从设计到数据的 3–5 个敌人原型**
  - 从 `enemies_life_events_v1_index.csv` / `LIFE_EVENTS_ENEMIES_V1` 中，确定 Demo 期的 **3–5 个代表敌人**：
    - 普通敌人 2–3 个（如：昨天的 DDL、拖延菌群、财务大妈、闹钟…）。
    - 精英/小 Boss 1–2 个（如：查重红条、甲方的折磨、论文开头退错了）。
  - 为每个代表敌人写一份 **正式原型表**（可以基于 `TEMPLATE_ENEMY_PROTO`）：
    - HP / 颜色 / 角色定位。
    - 2–4 个技能，对应 `INTENTS_CATALOG` 里的意图条目。
    - 粗略数值带（普通/精英层级）。
  - 把这些原型映射到 **JSON / 数据文件**，接入当前 Enemy/Intent 系统。

- **Gap 2：一条“Demo 专用甘特路线”**
  - 设计一条很短的战役路径，例如：
    - 开局 → 普通战斗 → 事件或休息 → 普通战斗 → 组会 → 精英战/小 Boss → 结束结语。
  - 在数据/配置里为这条路线单独标记（方便 Demo 模式直接跳用）。
  - 确保：
    - 每个关键节点都能顺利进入/退出；
    - 不会卡在尚未完成的系统（例如尚未正式启用的大型剧情）。

- **Gap 3：组会的 Demo 级别体验**
  - 为 Demo 锁定一版最小可玩集：
    - 免费升级 ×1（可使用一次，逻辑完整）。
    - 升级/删牌 流程走通（UI + 持久化）。
    - 购卡 → 通过 RewardService 弹出一次简单卡牌选择。
  - Meeting 事件：
    - 从现有导师/助教事件中选出 1–3 条，接入 `MeetingEventModal`。
    - 使用当前 Condition/Effect 体系的最小子集（灵感增减、新任务线标记等）。

- **Gap 4：5–8 条“真的会出现在 Demo 里”的事件**
  - Gantt 气泡事件：
    - 挑选 3–5 条纯风味或轻效果事件（黑色幽默段子），挂在甘特气泡系统上。
  - Meeting 事件：
    - 与 Gap 3 复用/合并，确保 Demo 至少会遇到 1–2 次有选择的组会对话。
  - 这些事件不需要复杂长链，只要：
    - 看起来有梗；
    - 选项有清晰差别；
    - 结果在数值/卡组上有一点点实际反馈。

- **Gap 5：一轮“Demo 视角”的整体检查**
  - 数值层面：敌人 HP/伤害、卡牌强度在 Demo 难度下不过分离谱。
  - 体验层面：从开始游戏到 Demo 结束，**没有明显“这里是半成品”的断层**。
  - 表达层面：关键节点的文字（敌人名称、事件标题、选项文案）与系统行为一致，不误导玩家。
