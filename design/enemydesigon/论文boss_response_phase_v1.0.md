## 论文线 Boss · 阶段3：Response Letter 设计稿 v1.0

> 版本口径：只描述阶段3“写回信”这一段的体验与规则骨架，不涉及实现细节与具体数值。  
> 前置条件：阶段1、2 的战斗已经结束，玩家与论文从“对抗期刊系统与理想球”的战场退场。

---

### 2026.01.08 确定事项（口径锁定）

> 这是一份“设计决议记录”，用于统一后续文档与实现口径。

- **1) 灵感掉落**  
  - 确定：**大小战斗都会掉落「灵感」**。  
  - 灵感是一个奖励对象，**初始携带若干正面/负面标签**（混合存在）。

- **2) 战后标签编辑（选择奖励）**  
  - 确定：战斗结束后，玩家在奖励界面可以对灵感/论文的标签进行**添加或去除**（以 n 选 1/多选一的奖励形式呈现）。

- **3) 负面标签的功能（MVP：L1 低耦合）**  
  - 确定：负面标签会让论文 Boss **变得更强大**。  
  - MVP 口径：先采用 **L1 聚合**——Boss 强度只读取**负面标签数量**（不逐条耦合到具体负面内容），后续可按人力升级到 L2/L3（类别/严重度）。

- **4) 正面标签的语义**  
  - 确定：正面标签代表论文的“血量/生命线”（论文是否还“像一篇能投的稿子”）。  
  - 可选：正面标签也可以提供少量战斗能力/被动收益，但优先级低于“作为生命线”的语义表达。

---

### 0. 设计目标与整体定位

- **非战斗收尾段落**  
  - 阶段3不再是战斗或资源博弈，而是一段**短篇剧情 + 轻量检定**。  
  - 关键词：**深夜、写信、忐忑、自我怀疑、偶尔的一点点底气**。

- **标签闭环（主货币必须真正发挥作用）**  
  - 论文在前序战斗中积累 / 失去的**正向标签**，必须在这里真实参与结果计算：  
    - 决定你有哪些“底气”可以写进回信；  
    - 决定你在面对关键问题时，成功“自圆其说”的概率与骰池。  
  - 玩家不需要再操作标签本身，标签只作为**检定加值 / 额外骰子来源**出现。

- **DND / CoC 风格的“检定 + 宿命感”**  
  - 玩家在表层只做一件事：**选择“应对策略 / 态度”**。  
  - **成功率与额外骰子由“论文标签 + 主角特质”组成**，并以跑团风格的文案展示给玩家。  
  - 通过“掷骰 + 结果旁白”，表达那种：  
    > “我已经尽力，但命运并不完全掌握在我手里”的感觉。

---

### 1. 场景与演出框架

- **入场演出**  
  - Boss 战 UI 淡出，切到一张静态但有细微动效的场景：  
    - 深夜房间、电脑屏幕、论文纸堆、咖啡/水杯。  
    - 屏幕上已经打开了审稿意见与回复编辑界面（Journal 系统）。  
  - BGM 从战斗曲转为低沉/空旷的环境音（键盘声、空调、电流噪音）。

- **界面布局（概念级）**  
  - **左侧**：2–4 条“最终问题（Final Questions）”列表。  
  - **右侧**：Response Letter 草稿区域（主要是文本展示，不做真实编辑器）。  
  - **底部/侧边**：  
    - 若干枚“论文关键标签”以小词条/便签形式贴在一侧（不需要可点击）；  
    - 一行简短的 Stress/精神状态描述（如：“你已经精疲力尽，但还撑得住几轮推敲。”）。

---

### 2. 输入：从战斗到“最终问题”

阶段3开始时，系统从前序战斗状态中抽取以下信息：

- **论文标签状态（来自整条论文线）**
  - 幸存的正向标签（尤其是：理论、方法、应用、表达/合规等轴上的关键标签）；  
  - 战斗中新获得的标签（Boss 掉落、事件奖励等）；  
  - 重要黑话标签（如“过度宣传”“样本不足”“立意不清”等）。

- **审稿球 / 敌方行为摘要**
  - 哪些理想球曾经放出大招、打掉过哪些类型的标签；  
  - Boss 行为日志中反复出现的批评轴（方法 / 积累 / 影响 / 规范）。

- **主角特质 / 状态（被动，不做额外 UI 操作）**
  - 少量与写作/沟通相关的“个人能力特质”，例如：  
    - `长文写作熟练`、`善于包装`、`倔强守原则`、`社交笨拙` 等。  
  - 当前精神状态的简化档位：**尚算冷静 / 极度疲惫 / 濒临崩溃**。

基于这些输入，系统生成 **2–4 条“最终问题（Final Questions）”**：

- 每条问题结构包括（概念层）：  
  - `type`: 问题类型（理论 / 方法 / 应用 / 表达 / 伦理 …）；  
  - `text`: 审稿人用语风格的最终问题文案；  
  - `severity`: 严重程度（影响权重）；  
  - `source`: 来源信息（哪颗球/哪类行为积累而来，仅用于 flavor）。

> 示例（概念文案）：  
> - “我仍不确定本工作的**理论贡献**是否足以支撑当前的叙事。”（type=理论）  
> - “本稿的**样本量与实验设计**似乎不足以支持所有结论。”（type=方法）  
> - “请更具体地说明这项工作在真实场景中的**应用价值**。”（type=应用）

---

### 3. 玩家交互：每条问题 = 一次“策略选择 + 检定”

对每条最终问题，玩家经历一轮相同的流程。

#### 3.1 选择“应对策略”（而不是选标签）

为每条问题提供 2–3 种**回答姿态 / 策略**（示例）：

- **【老实补洞】**  
  - 承认问题、用真实工作细节与补充计划来回应。  
- **【这是我们futurework】**  
  - 用圆滑语言、结构调整和引用来淡化/转移问题重心。  
- **【坚持立场】**  
  - 维护原有理论/叙事，承认有限度修补，但不愿大幅退让。

玩家只需要在这几种策略中**选一种**，不去挑具体标签。

#### 3.2 计算本次检定的“底气”和骰池

系统在后台基于以下内容计算检定参数：

- **与问题类型匹配的论文标签（正向）**  
  - 例：问题 type = 方法 时：  
    - `方法严谨` → +3  
    - `样本扎实` → +3  
    - `实验设计清晰` → +2  
  - 若某关键标签在战斗中被打掉，则对应加值缺失。

- **相关黑话标签带来的惩罚**  
  - 如 `样本不足`、`结果夸大` 等，根据问题 type 施加 -X 修正。

- **主角特质的被动修正**  
  - 示例：  
    - `长文写作熟练`：在【老实补洞】策略时 +1；  
    - `善于包装`：在【花式话术】策略时 +2；  
    - `倔强守原则`：在【坚持立场】策略时 +2（但可能对某些问题增加失败后果）。  
  - 这些特质**不需要玩家单独操作**，只在计算时参与。

- **问题自身难度**  
  - 对于 severity 高的问题 → 提高目标阈值 / 降低基础成功率。

**展示方式（概念级文案）**：

> “你准备以【老实补洞】的方式回应这个问题。  
>  
> 这次你能依靠的东西：  
> - 论文标签：  
>   - 理论严谨：+3  
>   - 参考文献充分：+2  
> - 个人特质：  
>   - 长文写作熟练：+1  
> - 问题本身的难度：-2  
>  
> **本次检定：掷 1d20 +4，目标值 13。**”

实现上可以自由选择“加值”或“额外骰子（优势）”的表达方式，只要语气是跑团式即可。

#### 3.3 掷骰与结果档位

- 玩家确认后，屏幕上出现一个简短的“掷骰动画”：  
  - 骰子数值 / 高低不必完全暴露，只展示结果档位即可（成功 / 勉强 / 失败，可选大成功）。  
- 内部将掷骰结果映射为三个（或四个）档位：  
  - `well_addressed`（充分回应）；  
  - `barely_addressed`（勉强说得过去）；  
  - `handwaved / failed`（基本没解决）；  
  - （可选）`critical_success`（大成功，用于极少数戏剧性场景）。

---

### 4. 文案与情绪：旁白承载 CoC / DnD 味道

对于每次检定的结果，给出一小段高密度旁白，承载“宿命感”与自我审判。

- **充分回应（well_addressed）示例**：

> “你在信里老老实实地写下补充实验的设计，没有夸张，也没有回避。  
> 当你读完这一段时，短暂地感觉自己并不是完全在撒谎。  
> 就算他们最后仍然拒稿，你知道这部分至少经得起一次质疑。”

- **勉强回应（barely_addressed）示例**：

> “字面上看，你已经给出了一个合理的解释。  
> 只是你很清楚，这些话大多是在已有文本上缝缝补补。  
> 如果有人真的顺着这段话往下问，你并没有多少底牌。”

- **失败 / 搪塞（failed）示例**：

> “你写下了一段礼貌得体的官话，把责任拆散在若干客观条件上。  
> 光标在句号后闪烁了很久，你试图改动几个形容词，却发现越改越空。  
> 最后你只是按下保存，假装自己已经‘做了能做的事情’。”
- **（可选）大成功（critical_success）示例**：

> “你意外地找到了那段被你忘在附录里的分析。  
> 当你把它整理进回信时，整封信的重心悄悄发生了偏移——  
> 你第一次产生了一种危险的想法：  
> 也许，这篇稿子其实真的值得被好好读一读。”

这些旁白不直接改变数值，只负责：

- 连接玩家在前序阶段的所有决策与情绪；  
- 让每一次检定都像是在审判“你和论文这几年到底做了些什么”；  
- 强化那种既现实又略带 CoC 味道的“命运感”。

#### 4.1 提交按钮后的短场景：导师的两句话

> 当你最终在界面上点下“Submit”时，浏览器没有任何特效，只是正常地刷新了一次页面。  
> 那一瞬间，你反而更累了——好像这几年全部的重量，都在那一声轻微的鼠标点击里泄了气。  
>  
> 你把电脑合上，办公室只剩下空调的嗡鸣。  
> 导师难得没有皱眉，只是看了你一眼，像是在确认什么，又像是在告一段落。  
>  
> “人类全部的智慧，大概就包含在这五个字里。”  
> “等待，和希望。”  
>  
> 他说这话的时候，没有承诺结果，也没有任何鸡汤。  
> 只是像一个已经经历过很多次的人，把这一次的等待，也轻轻放回到你手里。  
> 你忽然有点害怕那封邮件真正到来的那一天，却也隐约知道——  
> 无论结果如何，今晚这封回信，会一直留在你的记忆里。

### 5. 阶段输出：向战役层回传“投稿质量”

阶段3结束时，系统对所有问题的处理情况进行压缩，总结为一组简洁的结果数据：

- 对每条 `Final Question` 记录其处理档位：  
  - `well_addressed / barely_addressed / failed / critical_success`；  
- 结合全局标签质量（核心卖点剩余情况、黑话总量等），生成一个  
  **`submission_quality_tier`**，例如概念档位：  
  - `坚实但有缺陷`  
  - `勉强站住`  
  - `摇摇欲坠`  
  - `奇迹般地严谨`（对应某些大成功组合）。

这些结果不会立刻给出 Accept / Major / Reject，只作为战役层“审稿结果事件”的输入之一：

- 战役层在若干回合之后触发 `审稿结果` 事件：  
  - 读取 `submission_quality_tier` + 难度档位 + 少量随机因素；  
  - 决定是 Accept / Major Revision / Reject 等结局；  
  - 并在该事件中承接：黑色幽默演出、理想/SAN 长期影响、遗物/心魔发放等。

### 6. MVP 约束与复杂度控制

为避免系统过重，本阶段的实现遵守以下约束（首版 MVP 口径）：

- **问题数量限制**  
  - 玩家第一次遇到论文 Boss 时，仅生成 **2–3 条最终问题**；  
  - 高难度 / 后续周目可扩展到 4 条，但不再增加新的系统。

- **策略选项限制**  
  - 每条问题最多提供 3 种策略（老实补洞 / 花式话术 / 坚持立场），避免选择过载。

- **标签显示数量限制**  
  - 屏幕上展示的论文标签控制在 **6–8 个关键标签**：  
    - 核心卖点标签；  
    - 若干执行层标签；  
  - 其他属性以抽象描述呈现（例如“方法整体：中等偏上”），不进入检定 UI。

- **检定反馈粒度限制**  
  - 玩家只看到“本次检定的构成摘要 + 结果档位”，不展示具体公式。  
  - 完整数学逻辑保留在设计/实现层，方便后续调试与数值平衡。

- **阶段时长限制**  
  - 整个阶段3限定在 **1–2 分钟内完成**：  
    - 2–4 次“策略选择 + 掷骰 + 旁白”；  
    - 几行总结性旁白；  
    - 回到战役层的 `审稿等待` 任务块。  
  - 明确定位为战斗后的“短篇尾声”，而不是第二场主菜。

### 7. 后续可拓展方向（非 MVP）

以下内容仅作为未来版本的潜在拓展，不在 v1.0 中强制实现：

- **更丰富的策略分支**  
  - 为特定理想路线或极端 build 解锁额外策略选项（例如“完全躺平”“把责任推给领域本身”等）。  

- **更细致的个人特质系统**  
  - 将主角在其他线（教学、项目、生活）中的经历，折射到 response 检定中。  

- **多稿命运的长期记忆**  
  - 不同 `submission_quality_tier` 与最终审稿结果的组合，在后续年份事件中留下长期“学术履历印象”。  

这些拓展在设计上与本阶段骨架兼容，但需要更多 UI/数值/内容资源支撑，可在 Boss 2.x 之后逐步解锁。
