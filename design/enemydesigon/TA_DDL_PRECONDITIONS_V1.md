# TA DDL 精英前置机制评估 v1

## 目标

这份文档只回答一个问题：

**如果要实现 TA 线的 DDL 种植园主，现在还缺哪些最小中层能力？**

术语说明：

- 本文在 TA 线设计语义上，把玩家不断被迫处理的 `task` 统一称为“琐事”。
- 当前工程代码已经以 `chore` / `chore_host` / `publish_chore` 为主；文中出现 `task` / `task_host` / `publish_task` 时应视为兼容别名。

本轮不直接实现 DDL 精英，也不顺手把 TA 全琐事系统做完。目标是把它拆成：

- 已支持
- 需要薄扩展
- 需要先补中层

这样后面我们就能决定是“下一步直接做”，还是“先补宿主再做”。

## 设计核心

根据当前设计稿，DDL 种植园主的核心体验不是普通高数值精英，而是：

1. 强制压缩琐事处理窗口
2. 让未完成琐事数量直接转化成高伤害压力
3. 持续发布新琐事，让玩家始终处于追 deadline 的节奏里

也就是说，它不是“一个会打得更痛的怪”，而是：

**TA 琐事系统的极端压力测试器。**

## 当前已经支持的部分

### 1. 战斗内琐事宿主

当前已经有：

- `CombatChoreHost`
- 最多 3 个琐事槽
- 共享倒计时
- 琐事完成后重置共享倒计时
- 倒计时结束时，对仍 active 的琐事统一结算失败

这说明：

- DDL 精英最核心的“琐事压力容器”已经不是从零开始

### 2. 琐事 resolution actions

当前已经有：

- `transform_enemy`
- `apply_enemy_buff`
- `publish_chore`

这意味着：

- DDL 精英可以通过琐事失败直接触发战斗压力
- 也可以通过琐事链继续发布新琐事

### 3. TA 内容链

当前已经有：

- TA CSV 编辑源
- TA 生成脚本
- 主运行链 JSON 产物
- QuestLoader 可加载 TA 遭遇

所以从内容链角度看，DDL 精英没有额外阻塞。

## 还不够的地方

### 1. 缺少“琐事数量 -> 精英伤害/意图”稳定映射

DDL 精英设计里一个核心点是：

- 未完成琐事越多，`催` 伤害越高

当前系统已经可以让琐事失败后产生 buff，但还没有一层清晰的 runtime 入口，让敌人意图稳定读取：

- 当前 active 琐事数量
- 或当前 failed 琐事数量

如果没有这层能力，DDL 精英就只能靠写死数值或临时标记，主题会打折。

**判断：需要薄扩展。**

推荐最小做法：

- 给 `CombatChoreHost` 增加只读查询：
  - `active_chore_count()`
  - `failed_chore_count()`
- 让敌人 intent 条件或 damage 计算能够读取这些统计

### 2. 缺少“倒计时固定压成 1”的 encounter 级覆写

DDL 精英的另一个核心是：

- 琐事倒计时始终为 1

当前琐事宿主已经有：

- `shared_countdown`
- `shared_max_countdown`

所以这不是大问题。  
真正缺的是一个明确的配置入口，让特定 encounter 或 enemy passive 能把琐事宿主默认值改成：

- `shared_max_countdown = 1`
- 完成琐事后重置时也回到 `1`

**判断：已接近支持，只需要 very thin 配置扩展。**

### 3. 缺少“持续发布新琐事”的规范用法

DDL 精英的设计不是单一琐事，而是：

- 持续加压
- 琐事结束后还会继续来

当前 `publish_chore` 已经能链式发布，但我们还没有建立：

- “每回合自动补琐事”
- “维持场上琐事密度”

这条规则。

**判断：需要薄扩展，但不一定要先做完整琐事池。**

推荐最小做法：

- DDL 精英每个固定回合或固定意图可以发布 1 个预定义琐事
- 不做随机池抽取
- 先做一组固定琐事模板

### 4. 缺少“未完成琐事数”驱动的动态伤害表达

当前 runtime intent 主要还是静态或简单条件分支。

DDL 精英要比较像设计稿，最好支持：

- 基础伤害 + (`active_chore_count * N`)

如果没有这层，就只能退化成：

- active_chore_count 达到某阈值后切更强 intent

这两者都能做，但体验不一样。

**判断：这是本精英实现里最值得先明确的一点。**

我的建议：

- 第一版不做真正的动态乘法公式
- 先用分档 intent：
  - 0 个未完成琐事
  - 1 个未完成琐事
  - 2+ 个未完成琐事

这样能在不补新公式系统的情况下，先做出 DDL 的压力感

## 不建议现在一起做的内容

为了让 DDL 精英尽快可做，这些本轮不应该一起绑上：

1. 完整 TA 全主题琐事池
2. 随机琐事抽取系统
3. 复杂琐事 UI
4. 琐事 DSL
5. 三端的离线/复活规则
6. campaign 正式接入

换句话说：

**DDL 精英不应该成为“先把 TA 全琐事系统做完”的借口。**

## 最小可做版本建议

如果下一步真做 DDL 精英，我建议它的第一版只依赖这 4 件事：

1. `CombatChoreHost` 可配置为共享倒计时固定 1
2. DDL 精英能每回合或固定阶段发布一个预定义琐事
3. DDL 精英的高伤 intent 能按未完成琐事数分档
4. 琐事失败仍然能通过现有 resolution actions 继续加压

这样就能做出：

- 倒计时极短
- 琐事不断压上来
- 未完成越多越痛

已经足够像 DDL 精英了。

## 推荐实现顺序

如果后面要真正实现 DDL 精英，建议顺序是：

1. 给 `CombatChoreHost` 增加琐事数量查询接口
2. 给 TA encounter/chore host 增加 `shared_max_countdown = 1` 配置能力
3. 给 DDL 精英做“按 active chore 数分档”的 intent
4. 给 encounter 配一条固定琐事发布链
5. 最后再补更复杂的琐事池和演出

## 结论

DDL 种植园主现在**不是完全不能做**。

它已经具备的地基有：

- 共享琐事宿主
- 琐事链式发布
- TA CSV -> JSON 主运行链

它还缺的不是“大系统”，而是三层很小的中层能力：

1. 琐事数量查询
2. 倒计时固定为 1 的 encounter 级配置
3. 基于未完成琐事数的分档伤害/意图

一句话总结：

**DDL 精英已经进入“可以开始实现前置能力”的阶段，但还不建议直接跳到完整精英实现。**
