# TA 敌人可实现性映射 v1

## 目标

把 [TA-2_extracted](/D:/PHD_SIMULATER/docs/design/enemydesigon/TA-2_extracted) 里的助教线设计，压成一份真正能指导实现顺序的落地文档。

这轮只回答 4 个问题：

1. 哪些机制当前系统已经支持，可以直接复用
2. 哪些机制只差一层薄扩展，可以在普通开发节奏里补上
3. 哪些机制当前没有宿主，必须先补中层系统
4. 哪些敌人应该先做，哪些敌人应该后做

术语说明：

- 本文在 TA 线设计语义上，把玩家不断被迫处理的战斗内 `chore` 统一称为“琐事”。
- 当前工程代码统一使用 `chore` / `chore_host` / `publish_chore`；文中若出现旧的 `task` 说法，应视为历史命名，而不是当前实现接口。

本轮不做：

- 最终数值
- 正式敌人配置文件
- 琐事系统正式实现
- 美术和演出接入
- 战斗平衡结论

## 设计总览

助教线不是一组零散敌人，而是一整条共享玩法线。

核心体验是：

- 高频琐碎压力
- 琐事倒计时
- 失败后滚雪球
- 多单位协同带来的目标优先级压力

当前设计包含：

- 共享琐事系统
- 四个普通主题：答疑、点名、交作业、小组考核
- 两个精英：DDL 种植园主、三端（微信/QQ/邮箱）

其中最重要的不是单个敌人的技能，而是“琐事系统 + 敌人行为”一起制造压力。

## 当前战斗系统能力判断

### 已支持

这些能力当前系统已经存在，基本可以直接复用：

1. 敌人意图与被动
- 敌人有 intent 和 passive 结构
- 支持按回合执行、条件执行、广播提示

2. 基础伤害 / 格挡 / buff / debuff
- 普通成长、给自己加状态、给玩家上状态都已有宿主

3. 多敌人战斗与多目标交互
- 多单位战斗、敌人预告、目标选择都已存在

4. 队列 / 指针 / 位移类系统
- 这对部分白色风格琐事、位置控制类设计有帮助

5. 一部分回合内统计
- 当前系统已经有类似“本回合打出特定类别牌”的统计字段
- 说明继续补更多回合计数不是从零开始

### 部分支持

这些能力有一部分基础，但还不能直接当成现成功能使用：

1. 条件型出牌统计
- 例如同色牌数量、交替打不同色牌、连续若干次某类行为
- 当前可以补，但不是现成能力

2. 多单位协同触发
- 例如队友受伤时主怪获益
- 当前可以通过被动 / 事件响应补出来，但需要为敌人侧整理清楚触发入口

3. 手牌干扰与节奏打断
- 系统里已有抽牌、弃牌、状态，但具体到“助教搭子”这类干扰还要确认实现方式

4. 按当前 HP 比例结算的敌人输出
- 不复杂，但需要整理成明确效果类型，而不是临时写死

### 仍欠缺

这些能力当前仍没有完整宿主，或虽有基础但还没整理成可直接复用的完整机制：

1. 琐事系统的更高阶表达
- 当前战斗里已经有正式的 `chore_host` 宿主
- 已支持“场上最多 3 个琐事、共享倒计时、完成重置倒计时、失败结算”的最小闭环
- 但更复杂的条件型琐事、通用 UI、和更丰富的结算 DSL 还没有补齐

2. 琐事完成 / 失败事件链
- 例如“任意琐事完成时重置倒计时”
- “琐事失败后对敌人施加群体增益”
- 当前已有 resolution action 基础，但更通用的事件流和广播整理还不完整

3. 敌人本地积压副资源
- 新版三端的核心不再是离线复活，而是每个端口维护一个会持续增长、会被玩家伤害削减、并直接转化为伤害的 `积压` 计数
- 当前系统可以用 enemy passive / counter 逼近，但还缺一层更清晰的敌人侧副资源表达与意图读数

4. 琐事压缩型精英规则
- DDL 种植园主直接把倒计时压到 1，并按未完成琐事数放大伤害
- 这建立在琐事系统已经落地的前提上

## 机制映射结论

### 机制表

| 机制 | 当前支持度 | 说明 |
| --- | --- | --- |
| 基础攻击、防御、成长 | 已支持 | 可直接通过现有敌人 intent / passive 表达 |
| 多敌人协同 | 部分支持 | 需要明确敌人侧事件响应入口 |
| 手牌干扰 | 部分支持 | 需要确认敌人行为和玩家手牌系统的接缝 |
| 回合内出牌统计 | 部分支持 | 有基础，但离琐事条件还差一层抽象 |
| 按 HP 比例结算伤害 | 部分支持 | 可以做，但最好抽成明确效果 |
| 琐事倒计时系统 | 已支持 | 当前已有 `chore_host` 最小闭环 |
| 琐事完成/失败广播 | 部分支持 | 已有 resolution actions，但更通用事件流仍需整理 |
| 敌人本地积压副资源 | 部分支持 | 可用敌人局部计数/状态实现，但最好整理成明确的敌人侧副资源读口 |
| 倒计时压缩型精英规则 | 部分支持 | 基础宿主已就位，仍需整理成更完整的精英规则 |

## 按主题映射

### 一、答疑主题

**目标体验**
- 高频、琐碎、解释成本高
- 不是一下很痛，而是不断打断你的节奏

**敌人构成**
- 好奇宝宝
- 助教搭子

**关键机制**
- 基础攻击 / 格挡 / 成长
- 完成琐事后触发成长
- 干扰玩家手牌或节奏

**系统支持度**
- 好奇宝宝：大体已支持
- 助教搭子：部分支持

**判断**
- 这是最适合第一批实现的主题之一
- 不依赖完整琐事系统也能先做出基础敌人版

### 二、点名主题

**目标体验**
- 时间压力
- 有限时优先级目标

**敌人构成**
- 迟到学生
- 代到学生
- 拖延怪

**关键机制**
- 倒计时前必须处理的目标
- 拖延 / 垃圾注入 / 回合压力
- 琐事失败惩罚更强

**系统支持度**
- 当前已有正式琐事宿主，可做最小 deadline 版本
- 如果要做更丰富的失败惩罚和链式压力，仍需要继续扩展

**判断**
- 适合第二批，不适合第一批直接完整落地

### 三、交作业主题

**目标体验**
- 资源膨胀
- 越拖越痛
- 必须及时控场

**敌人构成**
- 卷王
- 作业堆

**关键机制**
- 成长型主怪
- 按当前生命或资源规模放大压力
- 目标控制与优先级选择

**系统支持度**
- 高
- 这组敌人大多数都能用现有状态和数值结构表达

**判断**
- 这是最适合第一批实现的主题

### 四、小组考核主题

**目标体验**
- 多单位协同
- 主怪吃队友收益
- 玩家要做击杀顺序决策

**敌人构成**
- 逼王
- 普通学生

**关键机制**
- 队友受伤或死亡触发主怪收益
- 主怪与普通学生之间的资源关系

**系统支持度**
- 普通学生：高
- 逼王：部分支持

**判断**
- 可以较早实现，但逼王需要敌人侧协同触发整理

### 五、DDL 种植园主

**目标体验**
- 把助教线琐事压力压成极端版本
- 倒计时极短，失败惩罚直接压到玩家脸上

**关键机制**
- 琐事倒计时压缩到 1
- 按未完成琐事数量提高压制伤害

**系统支持度**
- 中低
- 基础琐事宿主已存在，但“倒计时压到 1 + 按未完成琐事数放大压制”仍需专门整理

**判断**
- 应放到后期实现
- 更适合作为琐事系统落地后的精英验证对象

### 六、三端（微信/QQ/邮箱）

**目标体验**
- 三目标战
- 放着不管就越积越痛
- 玩家需要判断优先处理与优先击破顺序

**关键机制**
- 各端独立积压计数
- 玩家伤害会等量处理对应端积压
- 敌方行动时按剩余积压结算伤害

**系统支持度**
- 中
- 普通三目标战已支持；还需要一层敌人侧积压副资源与意图读数

**判断**
- 新版比旧版离线复活清楚很多
- 不再是重中层阻塞，但仍建议在基础 TA 骨架稳定后再做精英调参

## 敌人实现等级

### A：可以直接做
- 普通学生
- 好奇宝宝
- 卷王
- 作业堆

这些敌人的共同点是：
- 主要依赖基础伤害、防御、成长、目标控制
- 不要求全新的琐事宿主
- 适合作为助教线第一批可玩的骨架

### B：需要少量补机制
- 助教搭子
- 逼王
- 拖延怪
- 三端（微信/QQ/邮箱）

这些敌人的共同点是：
- 已有系统能覆盖大半
- 但需要补一层敌人协同触发、手牌干扰、或敌人侧副资源/结算效果整理

### C：需要先补中层机制
- 迟到学生
- 代到学生
- DDL 种植园主

这些敌人的共同点是：
- 明显依赖琐事系统、倒计时结算或更强的战斗事件宿主
- 如果当前直接写，会把实现做成大量特判

### D：暂不建议先做
- 当前没有必须放进 D 档的普通敌人

## 推荐实现顺序

### 第一批：先做可直接落地的骨架敌人
- 普通学生
- 好奇宝宝
- 卷王
- 作业堆

**原因**
- 主题感已经足够强
- 系统支持度高
- 可以最快产出第一批助教线敌人原型

### 第二批：补薄机制后做中层敌人
- 助教搭子
- 逼王
- 拖延怪
- 三端（微信/QQ/邮箱）

**原因**
- 需要少量敌人协同或干扰机制
- 但还不需要完整琐事系统
- 三端新版只需要敌人侧积压计数与精英数值调参

### 第三批：琐事系统落地后再做琐事驱动敌人
- 迟到学生
- 代到学生

**原因**
- 这两者更依赖“限时不处理就出事”的结构
- 如果没有琐事宿主，体验会打折很多

### 最后一批：精英收尾
- DDL 种植园主

**原因**
- 它们最能代表助教线的系统表达
- 但也最依赖前置机制成熟
- 放最后做，风险最低，完成度也更高

## 明确的实现前置

在真正开做 TA 线之前，建议先决定以下前置是否成立：

1. 是否允许第一批普通敌人先脱离完整琐事系统独立上线
2. 是否要先补一个最小琐事宿主，而不是直接追求完整琐事系统
3. 是否接受 DDL 精英明确延期到后面批次，并把三端按新版积压机制单独评估

如果答案分别是：
- 可以
- 可以
- 接受

那么 TA 线就已经进入可执行状态。

## 当前结论

这组设计不是“还不能做”，而是：

- **普通敌人已经可以开始做**
- **主题核心系统还需要一层中层宿主**
- **精英设计应放在后期实现**

因此，最合理的现实路线不是“一次做完整条助教线”，而是：

1. 先做第一批骨架敌人
2. 再补中层琐事机制
3. 最后做精英和复杂主题敌人
