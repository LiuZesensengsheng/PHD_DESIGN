# TA 敌人可实现性映射计划 v1

## 目标

把 [TA-2_extracted](/D:/PHD_SIMULATER/docs/design/enemydesigon/TA-2_extracted) 里的助教线设计整理成一份可直接指导后续实现排期的映射文档。

这轮不直接做敌人，不做数值，不做美术接入。只回答三个问题：

1. 哪些设计现在系统已经能做
2. 哪些设计需要补机制
3. 哪些敌人应该先做，哪些应该后做

## 范围

本轮分析覆盖：

- 任务系统
- 四个普通主题：答疑、点名、交作业、小组考核
- 两个精英：DDL 种植园主、三端（微信/QQ/邮箱）
- 各主题敌人行为描述
- 这些机制与当前战斗系统的对应关系

本轮不做：

- 最终数值
- 正式敌人配置文件
- 敌人正式实现
- 玩法重设计
- 文案润色

## 输出物

最终至少产出以下内容：

1. 助教线整体机制概览
2. 每个主题的机制映射表
3. 已支持 / 部分支持 / 未支持 机制清单
4. 每个敌人的实现等级
5. 推荐实现优先级

## 实施步骤

### 第一步：抽取机制列表

先把助教线设计里的核心机制按“机制类型”整理，而不是按文件散着看。至少整理出：

1. 任务倒计时机制
2. 任务完成条件机制
3. 失败惩罚机制
4. 高频低伤敌人节奏机制
5. 多单位协同机制
6. 目标优先级机制
7. 精英特殊规则机制

目标是把设计稿转成可对照系统能力的“机制清单”。

### 第二步：对照现有战斗系统能力

把每条机制按当前系统支持度分成三类：

- 已支持：现有系统基本可直接复用
- 部分支持：已有一半能力，但需要薄扩展
- 未支持：当前没有，需要新增系统能力

重点对照：

- 敌人行为系统
- 回合 / 事件统计
- 队列 / 指针 / 位置系统
- buff / debuff / 状态系统
- 任务或 mission 类对象是否已有宿主

### 第三步：逐主题映射

建议按下面顺序分析：

1. 答疑
2. 点名
3. 交作业
4. 小组考核
5. DDL 种植园主
6. 三端（微信/QQ/邮箱）

每个主题都写清：

- 目标体验
- 敌人构成
- 关键机制
- 当前系统支持度
- 主要缺口
- 推荐实现顺序

### 第四步：给敌人打实现等级

每个敌人打一个落地标签：

- A：可以直接做
- B：需要少量补机制
- C：需要先补中层机制
- D：暂不建议先做

这一步直接服务后面的实现排期。

### 第五步：给出推荐实现顺序

最终输出三层建议：

1. 第一批先做：机制简单、主题感强、系统要求低
2. 第二批再做：需要少量补机制，但不算重构
3. 最后再做：机制复杂、精英级、容易牵动系统

## 初步优先级判断

基于当前设计文本，初步判断如下：

### 第一批优先候选

- 普通学生
- 好奇宝宝
- 卷王
- 作业堆

原因：

- 多为单体或双体敌人
- 机制集中在成长、常规收益、目标控制
- 对新系统要求较低

### 第二批候选

- 助教搭子
- 迟到学生
- 代到学生
- 逼王

原因：

- 更依赖手牌干扰、倒计时、协同收益、条件触发
- 可能需要补充统计或任务关联机制

### 最后再做

- DDL 种植园主
- 三端（微信/QQ/邮箱）

原因：

- 都是精英级设计
- 系统表达复杂度更高
- 更适合放在普通敌和任务系统落地后再做

## 风险点

本轮调研时要重点留意：

1. 设计很强，但当前系统没有宿主，尤其是任务系统本身
2. 任务条件可能依赖战斗事件统计，例如同色牌数量、交替出牌等
3. 精英机制容易过早拉高系统复杂度，导致普通敌实现被反向拖慢

## 完成标准

这轮算完成，需要满足：

1. 有正式映射文档
2. 每个主题都被分类分析
3. 每个敌人都有实现等级
4. 有已支持 / 待补 / 未支持清单
5. 有明确推荐实现顺序

## 默认执行顺序

如果明天直接开始，建议顺序是：

1. 先通读 TA 设计并抽机制
2. 再对照现有战斗系统
3. 再写映射文档主体
4. 最后补实现优先级建议
