## 你不用看机器编号

这份包只给你看人能判断的场景。每条只有一个短审核编号，比如 `04-01`。真正的追踪编号放在机器队列里，给 Codex 后续生成 decision shard 时使用；你审核时不用读，也不用关心。

你真正要判断的是：

1. 这个证据边界场景能不能想明白。

2. 这个边界是否值得以后继续收集证据或写进审核习惯。

3. 这条是不是已经越界成正式规则、默认生成、hard gate、learned 或 reranker 行为。

## 怎么填

每条只填：

```text
决定:
信心:
备注:
```

决定可以填字母，也可以填中文：

| 选项 | 中文决定 | 后续处理 |
| -- | -- | -- |
| A | 先收下 | 值得成为后续 reviewed seed 候选 |
| B | 需要更多证据 | 方向有用，但还缺具体例子或反例 |
| C | 拒绝过度推断 | 说得太大、不安全，暂时不能提升 |
| D | 合并重复 | 和别的条目太像，应该合并 |
| E | 无法判断 | 这条对人不够清楚，先打回重写 |

信心可以填：高 / 中 / 低。

## 重要边界

下面的例子都只是为了帮助审核，不是正式卡牌设计，不是正式敌人数据，不会进入运行时，也不会改变现有 UI、存档、capability graph、report-only registry、hard gates、默认生成、learned 或 reranker 行为。

这份审核包本身也不会提升证据等级。所有 source case 仍然保持：

```text
design_note / human_curated / review_needed / advisory_context_only
```

只有你明确审核后，我才会另建 decision shard。现在这份包只是待审队列。

## 术语小字典

* 证据追踪: 一条结论能不能说清楚来自哪里、还有哪些不确定。

* 权限 wording: 文档里是不是清楚写明“只能建议，不能决定”。

* 消费者范围: 哪些报告或工具可以读这条证据，哪些不能读。

* 隔离: source-mined 或 generated 线索不能自动变成 reviewed 证据。

* feature hint: 帮后续分析分桶的轻量标签，不是正式规则。

* negative control: 明确记录“这里不应该成立”的反例。

* feature projection: 把 case 翻译成报告可用特征，但不改变权限。

* report-only handoff: 把信息交给报告使用，但报告仍然只是建议。
