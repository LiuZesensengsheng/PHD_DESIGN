## Key Outcomes

- The recommended migration order is now:
  1. `Combat Action Contracts + Queue Skeleton V1`
  2. `Task Resolution Orchestration Cutover V1`
  3. `Card Play Orchestrator Entry Cutover V1`
  4. `High-Frequency Effect Planner V1`
  5. `Combat Post-Resolution Policies V1`
  6. `Turn Flow Orchestration V1`
- The repo now has an explicit planning stance:
  - short migration window is acceptable
  - long-term dual-track combat flow is not
- The first two combat-orchestration slices are now landed together:
  - queue skeleton exists
  - task-host resolution no longer relies on the old large branch interpreter in `CombatModel`
- Card-play mainline now also has a single orchestration home:
  - current trigger is still `CardPlayed`
  - current resolution ordering now lives in `CardPlayOrchestrator`
- Card-play orchestration is now no longer all-or-nothing at the effect layer:
  - simple high-frequency effects can already go through queue actions
  - unsupported effects fall back per effect, not per whole card
  - this keeps the migration window short without forcing a giant one-shot effect rewrite
- Combat post-resolution is now also more explicit:
  - card aftermath no longer depends on a grab-bag method body in `CombatModel`
  - combat-end cleanup and counter reset live behind one end-condition helper
  - route / off-color / end-check / secret-transform ordering is easier to see and test
- Turn flow now also has explicit orchestration seams:
  - enemy-turn start no longer inlines event publish + task tick logic in `CombatModel`
  - enemy-turn action step construction is now one builder seam instead of a hard-coded local branch
  - enemy-turn end no longer hides hook routing / prune / end-event / combat-end checks in one model method body
- Simple queued damage is now closer to legacy runtime parity:
  - planner and legacy effect no longer maintain separate simple-hit math
  - stable additive/multiplicative branches can move through queue actions with lower behavior drift risk
  - compatibility fallback is now concentrated more around side-effectful or multi-step damage cards, not basic damage arithmetic
- Queue coverage now includes a larger share of high-frequency attack cards:
  - repeated-hit attacks no longer force fallback just because they are not single-hit
  - ramping attack growth can happen as an explicit queued follow-up instead of hidden inside the legacy effect body
  - compatibility fallback is shrinking toward conditional reward / bespoke card gimmick cases
- V1 fallback boundaries are now cleaner:
  - card-level compat rewards no longer need to stay embedded in `EffectExecutor`
  - fallback still exists, but it is now more clearly “legacy effect body + orchestrated follow-up actions”
  - this reduces the amount of sequencing logic hiding in the old executor path
- More active cards are now fully on the planned path:
  - `dropkick` no longer falls back for damage + reward sequencing
  - `battle_trance` no longer falls back for the lock-after-draw sequence
  - the remaining fallback surface is now more concentrated in niche/complex effects rather than staple combat cards
- `combat orchestration v1` can now be treated as complete:
  - task resolution, card play, post-resolution, and turn flow all have explicit orchestration seams
  - most high-frequency combat effects now go through `plan -> queue -> drain`
  - old execution paths still exist as bounded compatibility fallback, not as the preferred place for new sequencing logic
