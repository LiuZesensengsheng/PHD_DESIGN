## Boundary Notes

- No runtime card data, formal cards, hard gates, default synthesis, learned/reranker
  behavior, or reviewed-evidence promotion changed.
- `sts1_exam_target_v1` is not a complete STS1 ontology.
- `card_package_variant_set_v1` recommends a backup for drafting, but it does not
  create complete card text or promote content.
- The STS1 exam result is an advisory simulation result only. It does not promote
  these draft cards into formal cards, runtime data, reviewed evidence, or hard gates.
- `card_package_draft_handoff_v1` does not generate complete card text. It only
  packages constraints, slot requests, prompt context, and validation commands.
- `sts1_four_character_exam_v1` is an exam harness over owner-supplied draft fixtures.
  It proves the loop can run for all four STS1 roles, but it does not prove autonomous
  LLM generation quality or balance.
- `llm_complete_card_draft_attempt_v1` does not call an LLM API or generate card text.
  It records and classifies an already-supplied draft attempt.
- `exam_iteration_run_v1` is the unit of learning for the report-only loop. It records
  what to patch next, but it does not promote cards or create hard gates.
- `exam_iteration_prompt_patch_proposal_v1` only recommends prompt and handoff
  patches. It does not call an LLM, generate card text, modify runtime data, promote
  formal cards, or enable learned/reranker behavior.
- `exam_iteration_batch_comparison_v1` only compares report-only iteration snapshots.
  It can say which failure families moved, but it cannot claim design quality,
  reviewed evidence, hard gates, or default synthesis readiness.
- `exam_iteration_generated_attempt_batch_run_v1` orchestrates already-supplied draft
  files through existing report-only surfaces. It does not call an LLM API, generate
  card text, promote formal cards, write runtime data, claim reviewed evidence, create
  hard gates, or enable learned/reranker behavior.
- `llm_draft_prompt_application_v1` records prompt patch advice and an externally
  supplied draft attempt. It does not call an LLM API, generate complete card text,
  alter the base handoff contract, promote formal cards, write runtime data, claim
  reviewed evidence, create hard gates, or enable learned/reranker behavior.
- Intake-linked generated-attempt batches still only orchestrate already-supplied
  drafts. They reject intake snapshots that claim runtime LLM calls, generated runtime
  card data, reviewed evidence, hard gates, default synthesis changes, or
  learned/reranker enablement.
- Generated-attempt negative controls are intentionally flawed speculative fixtures.
  They are not owner-reviewed evidence, runtime cards, formal cards, or learned
  training promotion.
