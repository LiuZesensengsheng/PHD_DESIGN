## Current Status

- The campaign branch is now synced with remote `master` as of `2026-04-21`.
- Reward opening in campaign no longer has a split host-level entry path.
- Reward internals still exist as two concrete services, but they are now hidden
  behind grouped reward ownership instead of being exposed directly on
  `CampaignState`.
- Thesis judgment is now stricter about its input contract:
  - submission flow must set the track tier before judgment runs
  - judgment no longer keeps a save-recovery compatibility path for missing tier
- Session-store readbacks are now stricter for the campaign fields that are
  fully self-owned by the current runtime write path:
  - malformed preserved values now collapse to empty/default shapes instead of
    being silently repaired on read
  - this reduces old-save preservation noise in startup hydration, trait reads,
    meeting progress recovery, temporary-debuff checkpoint restore, gossip seen
    state, deck persistence, and thesis-combat sync payload restore
  - setter-side canonicalization is now also largely removed; session-store now
    behaves much more like a strict typed boundary than a quiet repair layer
  - the remaining session-store flexibility is now mostly deliberate:
    - defensive copying for validated snapshot payloads
    - explicit business clamps where the rule itself is part of gameplay/runtime
- `CampaignState` is a little less like a hidden service locator than before:
  - return resolution no longer depends on a private resolver helper just to
    mutate board state
  - board mutation is still implemented by `startup_services.board_mutation`,
    but lifecycle callers now see a named host seam instead of a generic
    `_resolve_*` indirection
  - frame entrypoints now also use a direct named host seam:
    `frame_orchestrator`
- The next compat-layer scan should focus on:
  - non-session-store campaign compat surfaces such as remaining service-bundle
    aliases or transitional host/state seams
  - any remaining service-bundle aliases that are still only there for transition
    convenience rather than active shell contracts
  - the most likely next narrow slice is `_resolve_fusion_and_stabilize()` and
    any other remaining `_resolve_*` style helpers that still exist only for
    transition convenience
  - after that, the next worthwhile seam scan is the small lifecycle-machine
    resolver helpers inside `CampaignTurnOrchestrator` /
    `CampaignEndTurnOrchestrator`
