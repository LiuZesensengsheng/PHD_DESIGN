## Candidate Long-Term Memory

- `campaign_power_curve_model_v1` currently exists as a docs-only exploration surface
  under `docs/exploration/CAMPAIGN_POWER_CURVE_LAB.md`.
- The model is explicitly report-only and must not expose `overall_pass`,
  `hard_gates`, or blocking readiness verdicts.
- The first stable boundary is to express enemy pressure as a vector request before
  any monster numeric implementation exists.
- The first checkpoint suite now covers starter, build, pivot, and mature phases; late
  phase remains intentionally deferred until fail-state and anti-infinite wording is
  more stable.
- New checkpoints should start as `hypothesis_draft`; stronger evidence labels change
  advisory use, not pass/fail authority.
- `mechanism_online_timing` is phase-aware and report-only; `conditional_online` is an
  honesty label, while `over_online` is a review prompt, not a legality verdict.
- `online_timing_modifiers` separate final-shell coherence from route credibility:
  economy and compression can advance, delay, or condition an online claim, but cannot
  create pass/fail authority.
- Modifier-driven archetypes now exist for build payoff-only detection, pivot
  compression-route probing, and mature over-online texture probing; they are
  validation prompts, not monster stat blocks.
- `enemy_pressure_phase_band` now gives qualitative phase language for pressure
  intensity; it must not be interpreted as damage, health, enemy roster, or
  hard-counter guidance.
- `player_power_reason_code` is now the expected explanation layer for player vector
  scores; scalar-only output should be treated as incomplete report context.
- `deck_maturity_state` now has qualitative labels for assembly shape; labels such as
  `payoff_only` and `overfocused_brittle` are review prompts, not failure verdicts.
- `economy_state` now separates upgrade, removal, shop, event, transform, healing,
  opportunity-cost, and route-risk evidence; missing economy context stays unknown.
- `compression_state` now distinguishes persistent thinning from combat-only
  mitigation; discard filtering and draw selection are not persistent removal.
- `curve_checkpoint` now has a review checklist; checklist status is advisory and
  must not be used as implementation or balance authority.
- Starter/build/pivot/mature payload variants now show phase-specific contract usage;
  they are examples, not schemas or implementation requirements.
- The cardanalysis mapping draft ties curve fields to canonical report-only surfaces;
  it does not add a new CLI, fixture contract, default path, or authority layer.
