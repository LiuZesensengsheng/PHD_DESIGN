## Candidate Long-Term Memory

- Gossip/BBS social simulation is currently an exploration-only model. It emits
  structured state payloads, not final forum prose.
- The model should stay sidecar-shaped until a campaign/narrative integration point is
  explicitly chosen.
- Debunked rumor memory should keep both the original claim and the corrected status;
  do not erase the old rumor as if it never existed.
- `misunderstanding` is a separate path from `rumor_seeded`: low malice and low
  correction access keep damage repairable, while repeated behavior after clarification
  should escalate.
- `new_card_package_seen` accepts only public structured package signals in this lab;
  it must not call or imply ownership by `cardanalysis` internals.
- `public_achievement` can strengthen public success memory and slow decay, but it
  must not certify exact package identity or turn into reward/narrative execution.
- `faction_event` can create alignment drift, but unknown player consent must remain
  explicit so faction identity does not overwrite actor agency.
- `npc_conflict` should remain separate from `rumor_seeded` until an unverified factual
  accusation is present.
- `rumor_seeded` keeps heat separate from credibility and must not silently merge a new
  rumor into an old corrected claim.
- `private_leak` should quarantine boundary-sensitive material before any credibility
  transfer; the model should not store or generate private content.
- `scandal` can raise boundary heat while lowering rumor credibility; it must not imply
  that the attached rumor is true.
- `combat_result` can rebalance accessible memory toward performance, but it should not
  erase scandal memory or become achievement prestige by default.
- `boss_defeated` can split a clean milestone thread from an older scandal-tainted
  thread, but the parent link should preserve memory continuity.
- `rumor_debunked` backfire can lower credibility while briefly raising heat; backfire
  heat must not count as evidence for the rumor.
- `decay` should cool heat and reply pressure without deleting corrected rumor records,
  privacy boundaries, or milestone memory anchors.
- Reply pressure should allow silent memory and future hook readiness; not every
  interested actor should post immediately.
- The schema contract now treats `public_thread_update` and `public_thread_reply`
  as public visibility values that should attach to existing threads first.
- Minimal `PostState` and `ReplyState` records should store structured role/stance
  signals only, not final BBS prose.
- New gossip/BBS enum or status values should update both the contract and lab log
  in the same slice.
- Current code entrypoints for the sidecar are
  `contexts/gossip_bbs_social_simulation/contracts.py` and
  `contexts/gossip_bbs_social_simulation/transitions.py`.
- First-slice transition math is deterministic scaffolding, not tuned social
  simulation; keep fixture assertions focused on structure and invariants.
- `combat_result` should create `performance` memory and may reopen a thread,
  but it must not silently become `prestige`/milestone behavior.
- `boss_defeated` should split a milestone thread only when salience/split
  pressure are high enough, preserve the old thread through a parent link, and
  land package/milestone records through contract-owned `state_patches`.
