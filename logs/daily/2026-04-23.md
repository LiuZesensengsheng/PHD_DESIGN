## Current Status

- The trigger surface is no longer just a passive collector; it now has a real
  first consumer layer with explicit runtime ownership.
- Built-in campaign reactions are more centralized:
  - lifecycle machine still owns phase/step/window timing
  - task-area domain events still own factual board changes
  - trigger reactions now own stable reactive consumers
- This makes future trait/event work more AI-friendly because new reactions no
  longer need to choose between:
  - lifecycle hooks
  - raw `domain_events`
  - ad-hoc interrupt glue
- The next campaign architecture target is now clearer:
  - forced events should move to a narrow long-term path
  - but should not widen into a generalized interrupt-platform rewrite yet
- Forced-event Slice 1 is now landed in working tree:
  - explicit model exists
  - pending queue ownership exists
  - presentation now lives behind a dedicated presenter seam
  - trigger reactions now only detect and enqueue forced events
  - active and pending forced-event states are now explicit
  - gate retry/order behavior is now deterministic
  - the first extension guardrail now exists through the forced-event catalog hook
- The accepted narrow forced-event plan has now reached its intended V1 stop line:
  - explicit runtime owner
  - explicit presenter seam
  - deterministic gate/resolution contract
  - explicit catalog-backed extension seam for future producers
- The next campaign-timing cleanup has now started at the turn-loop contract
  layer instead of widening forced-event scope:
  - the system now has one explicit result object for a full turn cycle, rather
    than requiring callers to reconstruct the outcome from phase and interrupt
    flags
- The return-resolution path is now also on the same explicit-result direction:
  - startup/return callers no longer need to rely only on mutable lifecycle
    context fields to understand what happened during return processing
- Lifecycle reads are now also more explicit:
  - external readers can take one lifecycle snapshot instead of manually
    assembling `current_phase + last_turn_cycle_result +
    last_return_resolution_result`
- Startup is no longer the remaining implicit lifecycle entrypoint:
  - the concrete startup order still stays in `CampaignStartupOrchestrator`
  - but the top-level startup outcome now has an explicit lifecycle-owned
    result and snapshot surface
